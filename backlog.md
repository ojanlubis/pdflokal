- **[crit] The retention experiment is DEAD — do not read a result off it.** The campaign existed to produce ONE number: week-1 return rate of the paid cohort (vs April's ~0%). **70% of it ran blind:** Jul 7–11 (the GA4 blackout) took **291 of 416 clicks and IDR 33,206 of IDR 47,313.** The Jul 5–6 cohort (48 users) had its entire week-1 window fall across Jul 7–13, and 5 of those 7 days recorded nothing. On the 2 visible days, 0 of 48 returned — but that is a **lower bound from 2/7 of the window, not a finding.** Directionally consistent with April; it CANNOT confirm it. **Do not update beliefs on it. Re-run clean.**
  - Verified user identity survived the Jul-12 stream swap (sitewide returners exist on Jul 12: 4, Jul 13: 6), so the 0-of-48 is a real reading — just a badly truncated one.

- **[crit] The ads buy the right INTENT and show the WRONG AD.** Every keyword sits in one ad group ("Gabung PDF"), so people searching for compress/convert get a **merge** ad. From the search-terms report (Jul 7–11):
  | search term | match | clicks | impr | CTR |
  |---|---|---|---|---|
  | `png to pdf` | Broad | **0** | 365 | **0.00%** |
  | `kompres pdf 1 mb` | Broad | 1 | 209 | **0.48%** |
  | `jpg to pdf` | Exact | **0** | 164 | **0.00%** |
  | `kompres file pdf` | Broad | **0** | 155 | **0.00%** |
  | `gabung pdf dan foto` | Phrase | 7 | 167 | 4.19% |
  | `cara menggabungkan file pdf` | Exact | 7 | 163 | 4.29% |
  - Those 4 mismatched terms alone = **893 impressions, 1 click.** That is what drags mobile CTR to 1.20% (desktop 6.02%).
  - **We ALREADY have the pages for every one of them** — `/kompres-pdf-1mb`, `/jpg-ke-pdf`, `/kompres-pdf`. `kompres pdf 1 mb` is *literally* the SEO wedge (compress-to-fit-a-cap) and we're answering it with a merge ad.
  - Fix: **ad groups per tool, each pointed at its own SEO page.** Merge → `/gabung-pdf`, compress → `/kompres-pdf`, cap queries → `/kompres-pdf-1mb`, image → `/jpg-ke-pdf`.
  - ⛔ **CORRECTION (Jul 14, researched). An earlier version of this line said "negative-keyword `gabungkan file`" — THAT ADVICE WAS DANGEROUS, DO NOT FOLLOW IT.** A **broad** negative blocks any query containing ALL those words in ANY order ([docs](https://support.google.com/google-ads/answer/2453972)) — so `gabungkan file` would have blocked `gabungkan file pdf`, `cara gabungkan file pdf`, `gabungkan file pdf online`: **our only converting terms.** If you exclude the bare generic at all, it must be an **exact-match negative `[gabungkan file]`**.
  - ⚠️ **Negatives do NOT match close variants** — positives expand semantically, negatives are literal. **This is brutal in Indonesian**, which is agglutinative: `gabung` / `gabungkan` / `menggabungkan` / `penggabungan` / `digabung` are five separate tokens a negative list will not connect for you. Plan for a large, permanently-incomplete negative list; don't discover it.
  - **Better than negatives: fence the ad groups from each other** (compress terms negative in the Merge group, merge terms negative in the Compress group). That converts the 3,372-term sprawl from waste into routing. The other-tool traffic is GOOD — it's just landing in the wrong room.
  - **THE MECHANISM NOBODY SPOTTED: our homepage is a broad-match INPUT, not just a bad destination.** Google's matching reads *"the content of the landing pages"* ([docs](https://support.google.com/google-ads/answer/7478529)). Our homepage IS the editor and advertises merge + compress + convert all at once — so broad match read it, concluded "this advertiser does all PDF things," and matched accordingly. The per-tool SEO pages therefore fix the *matching signal* as well as the promise. Two birds.
  - Campaign matched **3,372 distinct queries** — but that count is NOT the evidence (broad match on head terms always has a long tail, and the report is privacy-thresholded so the true count is HIGHER). **The real evidence is the CTR column: 529 impressions on `png to pdf` + `jpg to pdf` → ZERO clicks**, vs 4.29% on `cara menggabungkan file pdf`. That is message-mismatch measured in our own account.
  - Google's status: *"Eligible (Limited) — Bid setting limited, Missing enough relevant keywords."* **"Bid setting limited" = a max-CPC bid limit is capping our bids** ([docs](https://support.google.com/google-ads/answer/6263057)) — see the bidding item below.

- **[✅ DONE Jul 14] Conversion tracking is LIVE.** Was `Conversions: 0.00` — never set up. Now: GA4 key event `download` → Ads conversion action **"PDFLokal (web) download"** (source GA4, **Primary**, **Count = One** per click, 90-day window). Goal **Engagement = account-default, 4 of 4 campaigns**. See Done section for the two traps this hit.
  - **BIDDING DELIBERATELY UNCHANGED** — still Maximize clicks. Let the Rp100k run die on Jul 17 without confounding it; switch bidding only on the NEXT campaign, once history has accumulated. This is *literally* Google's documented advice: *"If you have a brand new campaign, consider using Maximize Clicks to build traffic and conversion data first"* ([docs](https://support.google.com/google-ads/answer/7381968)).
  - **CORRECTION (researched Jul 14): "Smart Bidding needs 15–30 conversions/month" is a MISCONCEPTION as a blanket rule.** Reality: **Target CPA needs ZERO history** (*"Advertisers can start using Target CPA with no conversion history"*). **Maximize Conversions** has a *recommendation* of ≥15 in 30d, not a gate. Only **Target ROAS** enforces a hard minimum (15/30d on Search, values must be > 0 — which we don't have, since a free download has no revenue). Learning period is **up to 3 weeks / 1–2 conversion cycles**, not the "7 days" folklore.
  - Volume is a non-issue regardless: **~62 download conversions on Jul 13 alone.**
  - **Count = One is CONFIRMED correct** — Google's own framing is "every" for sales (repeat purchases add value), "one" for leads. The real test is *whether repeat events add value*: a user exporting 3 PDFs is not 3× the value to a free tool. With "Every," Smart Bidding would learn to chase multi-file power sessions.
  - **Primary alone is NOT sufficient** — biddable requires **Primary AND the goal is applied to the campaign AND the goal is biddable**. All three are now true. Note Google's doc warns *"conversion actions created through the Google Analytics interface will be set as secondary"* — ours was created via the Ads UI and landed Primary, but **always verify all three switches by hand.**

### ☠️ RESEARCHED Jul 14 — THE BLACKOUT'S REAL ROOT CAUSE (our recorded story was wrong)

- **[crit — the ID was never dead; we recreated the stream for nothing]** The five-day GA4 blackout was **not** "Google refused a corrupted measurement ID," and **not** the shared Ads tag. **`G-0XJY0WLCZ1` existed only as a DESTINATION of the Google tag `GT-5MRGDJVG`, never as a SOURCE — and a `G-` id that is only a destination cannot be loaded via `gtag/js?id=G-…`. That request 404s.** Measured live on Jul 14, two days *after* the "fix":
  ```
  gtag/js?id=GT-5MRGDJVG   → 200, 550,099 bytes — payload CONTAINS G-0XJY0WLCZ1   ← alive, today
  gtag/js?id=G-0XJY0WLCZ1  → 404, 1,584 bytes (HTML error page, not JS)
  ```
  **The fix was ONE LINE** — load `gtag/js?id=GT-5MRGDJVG`. Instead we recreated the stream, minted `G-7J8JF8XZ1Q`, swapped it across 4 pages and left the property with a second stream. **Unnecessary.** (Identity was not wiped — returners still exist post-swap — but it was a risk and a cost we didn't need.)
  - **⛔ NEW STANDING RULE: before declaring any measurement ID dead, curl the `GT-`/`AW-` container that carries it and grep for the `G-` id. If it's in there, the ID is fine and your PAGE is wrong.**
  - **Why GA4's own alarm never fired:** Tag Diagnostics escalates to "Urgent" only when a tag *"was previously detected but hasn't been detected in the last 48 hours."* Ours was reported as **never detected** — so the one native drop-to-zero alarm **structurally could not fire.** The worse the break, the quieter GA4 gets. **This is the argument for the external tripwire.**
  - **The "Data collection is active" badge is a BINARY PRESENCE CHECK** — one hit keeps it green, no volume awareness. A trickle of stragglers (browsers with the old container still cached) kept it green while collection was ~97% dead.
  - All of this is now in the **`google-measurement` skill** (user-level, so mesindev and GAS get it too).

### 🔬 RESEARCHED Jul 14 — where the Jul-13 ads audit was WRONG (read before acting on any of it)

- **[crit, 1-field fix] The bottom-of-page cause is a MAX-CPC BID LIMIT, not "Maximize clicks buys cheap clicks."** My original mechanism was backwards. Maximize clicks' documented FIRST priority is **spending the budget in full** — it will *raise* CPCs to do so; it is not a bid-suppression strategy and **has no position lever at all**. The real culprit is named in the campaign's own status: *"Bid setting limited"* = *"Maximum and/or minimum bid limits are preventing your bids from being fully optimized"* ([docs](https://support.google.com/google-ads/answer/6263057)). **Corroborating arithmetic: we spent Rp47,313 of Rp100,000 with 10 of 15 days gone — ~29% UNDERSPENT.** A Maximize-clicks campaign that *cannot spend its budget* is hard-capped by something. **ACTION: open the bid strategy and remove/raise the max CPC limit.** If we want the top slot deliberately, the purpose-built tool is **Target Impression Share (Absolute top)**, not Maximize clicks.

- **[crit — the strategic question that outranks all the tactics] A conversion is worth Rp0 to us, so every paid click is negative ROI by construction.** PDFLokal is free, client-side, has no revenue, and ads-as-monetization is ruled out forever. Paid search is a machine for buying users at a price — **we do not yet have a price we can pay.** No account restructure fixes that. **The honest reframe: the Rp100k did not buy users, it bought KEYWORD RESEARCH — and it has already paid for itself.** Route the demand it revealed into the SEO pages, where the traffic is free, permanent, compounding, and needs no conversion value to justify itself.

- **[high — NEW, and it changes the SEO pages] INDONESIANS SEARCH FOR CONVERSION IN ENGLISH.** `png to pdf` (365 impr) + `jpg to pdf` (164 impr) = **529 impressions, outdrawing every Indonesian merge term combined (163)**. Our SEO pages are Indonesian-titled throughout (`/jpg-ke-pdf`, h1 "JPG jadi PDF"). We had no way to know this from an Indonesian-only keyword assumption — the ad spend surfaced it. **Revisit titles/H1s/on-page copy on the conversion pages to carry the English forms too.** (Does NOT violate the page-earns-its-existence rule — same page, better copy, no new doorway pages.)

- **[high — NEW, affects the whole product line] "A to B" ↔ "B to A" is a documented close-variant failure, and file conversion is the WORST-hit category for it.** Exact match has not meant exact since ~2018: Google now matches reordered words, synonyms, paraphrases and "same intent" ([docs](https://support.google.com/google-ads/answer/9342105)). The reordering rule is blind to the fact that reversal FLIPS meaning in a conversion query (the classic "LAX to JFK" / "JFK to LAX" failure). **`jpg to pdf` was matched by an EXACT keyword — find out which one; if it was `[pdf ke jpg]`/`[pdf to jpg]`, that's the reversal bug, not a merge mismatch.** Every conversion ad group we ever build needs explicit negatives on the reversed form. Bracket-exact will not save us.

- **[med] "Quality Score" is NOT an auction input — stop treating it as a lever.** Google: *"Quality Score is not an input in the ad auction. It's a diagnostic tool."* ([docs](https://support.google.com/google-ads/answer/6167118)). Auction-time quality is computed fresh per query; the 1–10 number is a 90-day report card. Also: **the AD COPY is the first-order fix, the landing page is second-order** — a merge headline shown to someone typing `kompres pdf 1 mb` never earns the click, so the landing page is never even reached. My original framing had these backwards.

- **[med] Take the thematic-ad-groups half of modern advice and REFUSE the broad-match + Smart-Bidding half.** They're a package deal: Google says it is *"critical to use Smart Bidding with broad match"* ([docs](https://support.google.com/google-ads/answer/12159290)), and Smart Bidding needs conversion volume + budget we don't have. **At this budget the "outdated" config is the correct one: exact + phrase match, manual CPC, aggressive negatives.** Copying 2026 best practice wholesale here would be the most expensive mistake available to us. (One ad group per tool is a **STAG**, not a SKAG — that recommendation survives.)

- **[med] "We're winning the auction — 54% impression share, highest of anyone" was a BAD READ.** Auction Insights impression share is measured only on auctions *our* ads qualified for; Adobe's 24% has a completely different denominator ([docs](https://support.google.com/google-ads/answer/2579754)). Being #1 on IS in a keyword pool only we bid on **describes our keyword list, not our competitiveness.** The comparable metric is **Position Above Rate** — never pulled it.

- **[med] The mobile-CTR-is-caused-by-position theory is NOT established.** Competing explanation: **query mix**. 20,239 mobile vs 2,177 desktop impressions is a 9:1 skew no device split explains, and reeks of broad match dragging in low-intent mobile queries. **Falsification test: segment Device × Absolute-top-impression-rate.** If mobile abs-top ≈ desktop abs-top, position is NOT the cause and the theory collapses. We have one blended number and inferred a device-specific cause from it.

- **[med] We cannot learn anything from 7 clicks.** The best-performing term has 7 clicks. No restructure, bid strategy, or negative list makes that sample informative. **Do the restructure because Google documents it as correct — not because our data proved it. Our data proved nothing and cannot.**

- **[med] The "~95% of paid is mobile" premise is stale.** That's Ads 1 (April: 461 mobile vs 6 desktop new users). **Ads 2 is 58% mobile by clicks** (mobile 243 / desktop 131 / tablet 42). Mobile-first still holds directionally, but CLAUDE.md + `product-definition.md` quote 95% as if current. Mobile also takes **64% of spend (IDR 30,319) at 1/5 the desktop CTR**.

- **[med, still open]** Google Ads "Balance exhausted" warning. Campaign is nonetheless **still serving** (green, Eligible-Limited) with ~Rp53k left and 4 days to run. Decide: let it coast as a CPC/CTR learning run, or pause until conversion tracking + per-tool ad groups exist. **Recommendation: don't re-run until both are fixed.**
