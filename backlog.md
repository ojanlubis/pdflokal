- **[high]** Performance-optimization audit — likely the same root as the mobile canvas problem (memory pressure) — [page-rendering.js](js/editor/page-rendering.js), `memory/mobile-rendering.md`, `memory/canvas-performance.md`
  - User note (Jul 1): wants a dedicated performance pass, and suspects it correlates with the mobile canvas issue. It very likely does — same root cause. Per CLAUDE.md + memory, we made a deliberate tradeoff: **NO canvas eviction — pages render once and stay rendered** (evict→re-render caused white-flash flicker on mobile). Cost: **~4MB of canvas backing store PER page stays allocated for the whole session**. On a 10-page doc that's fine; on a 50–100 page PDF (this app targets files up to 50MB) that's 200–400MB+ of live canvas on a phone, which mobile browsers respond to by silently purging GPU backing stores → the exact flicker/blank-canvas behavior we keep fighting. So "fix the mobile canvas" and "optimize performance" are two names for the same memory problem.
  - The real fix both point to (already noted as future direction): **viewport-windowed rendering** — keep only the ~3 nearest pages as live canvases, downgrade the rest to a lightweight cached bitmap/thumbnail, re-promote on scroll. This is the "3-nearest viewport rendering" idea in `memory/architectural-direction-2026-06-09.md`. The reason plain eviction failed before was the evict→async-re-render gap flashed white; the fix is to swap to an already-decoded cached image (no async gap), not to a blank canvas.
  - Scope this audit to MEASURE first (don't fix blind): (a) instrument live-canvas count × bytes on a 50+ page doc on a real phone; (b) profile memory during scroll + pinch-zoom (pinch is already a known flicker offender — separate backlog item above); (c) time PDF.js render vs cache-restore paths. Bring numbers back before choosing between windowed rendering, off-thread render (Web Worker — item 2 in future-architecture.md), or a cheaper cache format. Read `memory/mobile-rendering.md` FIRST — it documents every eviction approach already tried and rejected, so we don't re-run a dead end.
  - Cross-links: pinch-zoom flicker item + mobile fast-scroll jump item above are both symptoms of this same rendering/memory layer.

- **[high]** Pinch-zoom flicker is severe on mobile — [page-rendering.js renderVisiblePages](js/editor/page-rendering.js), [zoom-rotate.js](js/editor/zoom-rotate.js)
  - User report (Jun 9, Android Chrome): pinch-to-zoom flickers very badly and uncontrollably.
  - Diagnosis (high confidence): zoom changes canvas dimensions → `pageCaches` entries become dimension-mismatched → `restoreCanvasFromCache` early-returns on the size guard → falls back to async PDF.js re-render via `renderVisiblePages` → produces visible blank-then-content states across all visible pages simultaneously. PR #66 doesn't help here by design.
  - Fix candidates (none cheap):
    - **(a) CSS scale during pinch**: apply `transform: scale(zoom)` to the page slots during pinch, only re-render the canvas buffer on pinch-end (the gesture itself is preview-quality, the commit is high-quality). Lower flicker risk; intermediate frames are scaled-pixel renders.
    - **(b) Double-buffer**: render to off-DOM canvas, swap when ready. Doubles memory during render.
    - **(c) Defer re-render**: only re-render once pinch settles (debounce). Final state is correct; during pinch, canvas stays at old zoom level visually until release. Trade-off: misleading "stale" state mid-gesture.
    - Author's lean: (a) — matches user expectation (pinch = preview, release = commit) and is the cheapest to ship.
