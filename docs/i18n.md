# Internationalization (i18n)

PDFLokal was built Indonesian-only. This document describes the **foundation** now
in the tree, the **conventions** for adding strings, and the **deferred plan** for
finishing the job. Read it before touching anything language-related.

> Status (Jul 2026): **Foundation only.** The `t()` layer exists and one file
> (`js/v2/install-prompt.js`) is fully migrated as a proof of concept. The rest of
> the app still holds inline Indonesian literals. The big extraction sweep is
> **deliberately deferred** ‚Äî see "Sequencing" below.

## What exists

| Piece | File | Role |
|---|---|---|
| Engine | `js/lib/i18n.js` | `t(key, params)`, locale detection, plurals, number formatting. Self-bootstraps on import. |
| Source dictionary | `js/locales/id.js` | Bahasa Indonesia ‚Äî the master copy AND the universal fallback. |
| Proof of concept | `js/v2/install-prompt.js` | Fully routed through `t()` (slots, step-list arrays, per-browser branches). |
| Formatting seam | `js/v2/download-sheet.js` `fmtMB` | Decimal separator via `formatDecimal()` instead of a hardcoded `,`. |
| Contract tests | `tests/core/i18n.test.mjs` | Headless proof of interpolation, plurals, fallback, number formatting. |

There is intentionally **no edit to `js/v2/app.js`** ‚Äî `i18n.js` self-initializes so
the app boot path is untouched. This keeps the foundation clear of the large
in-flight feature branch that rewrites `app.js` (see Sequencing).

## How `t()` works

```js
import { t } from '../lib/i18n.js';

t('install.cardSub', { screen: 'layar HP' }); // string with a {slot}
t('install.ios.steps');                        // ‚Üí array of strings
t('pages.deleted', { count: 3 });              // ‚Üí plural form by count
```

- **Lookup order:** active locale ‚Üí `id` fallback ‚Üí the key itself (a visible,
  greppable miss ‚Äî never a blank UI).
- **Slots, not substrings.** Anything that varies mid-sentence (the
  `hapemu`/`komputermu` device word, `N halaman`) is passed as a `{slot}`. Never
  splice a translated substring into a JS template literal ‚Äî word order differs by
  language.
- **Plurals** use `Intl.PluralRules`. A dict value can be `{ one, other, ‚Ä¶ }`,
  selected by `params.count`. Indonesian only needs `other`; English/Arabic/‚Ä¶ do
  not, so author plural keys as plural objects from the start.
- **Numbers:** `formatDecimal(n, digits)` / `decimalSeparator()` are locale-driven
  (`id` ‚Üí `0,5 MB`, `en` ‚Üí `0.5 MB`).

## Locale detection

`resolveLocale()` (in `i18n.js`), first match wins, all headless-guarded:
`?lang=` ‚Üí `/<locale>/` path prefix ‚Üí `localStorage['pdflokal-locale']` ‚Üí `<html lang>`
‚Üí `id`. Only locales with registered messages are eligible.

## Adding a string (today's rule)

1. Add the key to `js/locales/id.js` (nested object, dotted key, `{slots}`).
2. Call `t('your.key', params)` at the use site.
3. Keep the dictionary **honest**: no key that isn't actually read by `t()`.

## Adding a locale (later)

1. Create `js/locales/<locale>.js` mirroring `id.js`'s shape.
2. `registerMessages('<locale>', dict)` and add it to `LOCALES` in `i18n.js`.
3. `setLocale()` from a switcher, `?lang=`, or a path prefix.
4. SEO is separate ‚Äî see below.

## Sequencing (READ THIS BEFORE THE BIG SWEEP)

The work splits into two kinds, and only one of them conflicts with feature work:

- **(a) Foundation** ‚Äî new files + isolated seams. Near-zero merge conflict.
  **‚Üê this is what's done.**
- **(b) Extraction** ‚Äî rewriting ~28 files of inline literals to `t()`. Pervasive;
  `js/v2/app.js` is the worst. **High** conflict.

The `feat/edit-teks-asli` branch rewrites `app.js` by ~900 lines, plus `state.js`,
`render/*`, `format-bar.js`, and adds new modules with their own strings. Running
extraction (b) in parallel means resolving the same `app.js` conflict twice **and**
re-extracting the feature's new strings afterward. So:

1. **Foundation now** (done).
2. **Land `feat/edit-teks-asli`.** Don't block it on i18n.
3. **Then the extraction sweep** ‚Äî one pass over the settled `app.js` + all files,
   the feature's new strings included. Highest-traffic first: `app.js` ‚Üí
   `download-sheet.js` (rest of it) ‚Üí `page-manager.js` ‚Üí `celebrate.js` ‚Üí
   `changelog.js` ‚Üí `js/editor/*` ‚Üí `js/pdf-tools/*`. The invariant: with locale
   `id`, rendered output must stay byte-identical to today (the existing suites pin it).
4. **Second locale + SEO.** Landing-page copy lives in `seo/pages.json` and is
   generated by `scripts/gen-seo-pages.js` ‚Äî localize by extending that JSON to a
   per-locale shape and emitting `/<locale>/‚Ä¶` pages with **localized slugs** and
   **`hreflang`** alternates (none exist yet). Machine translation is worse than
   nothing here: the human Indonesian copy is a deliberate SEO moat ‚Äî each locale
   needs real native copywriting.

## Invariant for the whole migration

While `id` is the active locale, **the UI must render exactly the strings it renders
today.** `t('‚Ä¶')` for an `id` key returns the same bytes the literal did. Any diff in
shipped Indonesian copy during extraction is a bug, not a translation.
